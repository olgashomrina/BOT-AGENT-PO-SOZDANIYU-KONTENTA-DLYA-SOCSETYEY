# Деплой бота на Timeweb Cloud

Эта инструкция описывает, как перенести уже работающего локально бота
(см. `README.md`) на настоящий сервер, чтобы он работал круглосуточно, а не
только пока у вас на компьютере открыт PowerShell.

Инструкция написана для владельца без опыта администрирования серверов, но
часть шагов — это команды в терминале. Каждая команда сопровождается
объяснением простым языком, что она делает. Выполняйте шаги по порядку.

Бот работает в режиме **long polling** (сам опрашивает Telegram, а не ждёт
входящих запросов) — поэтому серверу **не нужен домен и не нужен SSL-
сертификат**. Достаточно самого дешёвого сервера с постоянным доступом в
интернет.

## Что понадобится перед началом

- Банковская карта РФ — оплата сервера на Timeweb Cloud идёт в рублях.
- Реальный токен бота от @BotFather и реальный ключ ИИ-прокси (те же, что
  описаны в `README.md`, шаг 4) — их нужно будет ввести **на сервере**, не
  здесь.
- Программа для SSH-подключения. На Windows 10/11 она уже встроена в
  PowerShell (команда `ssh`). Если команда `ssh` не находится — установите
  "OpenSSH Client" через Параметры Windows → Приложения → Дополнительные
  компоненты.

## Шаг 1. Создать сервер на Timeweb Cloud

Этот шаг вы делаете сами в личном кабинете Timeweb Cloud — он стоит
настоящих денег, и никто не может сделать его за вас.

1. Зайдите на https://timeweb.cloud/ и зарегистрируйтесь / войдите в личный
   кабинет.
2. Создайте новый облачный сервер (VPS/VDS) со следующими параметрами:
   - **Операционная система:** Ubuntu 22.04 LTS (или более новая LTS-версия,
     если 22.04 недоступна). Ubuntu выбрана как самый распространённый и
     хорошо документированный вариант для systemd-сервисов — по любой
     проблеме легко найти готовое решение в интернете.
   - **Конфигурация (CPU/RAM/диск):** минимальная из предложенных
     (обычно 1 vCPU, 1 ГБ RAM, ~15 ГБ диска). Бот не занимается тяжёлыми
     вычислениями сам — вся генерация текста идёт через внешний ИИ-прокси,
     поэтому мощный сервер не нужен. Если в процессе окажется мало памяти,
     конфигурацию всегда можно увеличить позже прямо в личном кабинете.
   - **Домен/SSL:** не нужны, пропустите эти опции при создании — бот
     работает через long polling, а не через веб-адрес.
   - **Публичный IP-адрес:** обязательно включите — он нужен, чтобы
     подключиться к серверу по SSH.
3. Дождитесь, пока Timeweb Cloud создаст сервер (обычно 1–5 минут). В
   личном кабинете появится IP-адрес сервера и данные для входа (логин,
   как правило `root`, и пароль либо SSH-ключ — в зависимости от того, что
   вы выбрали при создании).
4. Сохраните IP-адрес и пароль/ключ в надёжном месте (менеджер паролей) —
   они понадобятся в следующем шаге.

## Шаг 2. Подключиться к серверу и подготовить систему

Откройте PowerShell на своём компьютере.

1. Подключитесь к серверу по SSH, подставив ваш реальный IP-адрес:

   ```
   ssh root@ВАШ_IP_АДРЕС
   ```

   Это открывает удалённый терминал на сервере — дальше все команды в этом
   шаге вводятся уже там, а не на своём компьютере. При первом подключении
   SSH спросит, доверяете ли вы серверу — введите `yes`. Затем введите
   пароль (или, если использовали SSH-ключ, вход произойдёт автоматически).

2. Обновите список пакетов и сами пакеты системы:

   ```
   apt update && apt upgrade -y
   ```

   Это подтягивает последние версии системных программ и исправления
   безопасности — стандартный первый шаг на новом сервере.

3. Установите Python, инструмент для виртуальных окружений и Git:

   ```
   apt install -y python3 python3-venv python3-pip git
   ```

   `python3` — сам язык, на котором написан бот; `python3-venv` — то же
   самое виртуальное окружение, что и на Windows (см. `README.md`);
   `git` — нужен, чтобы скачать код проекта на сервер.

4. Проверьте версию Python (должна быть 3.10 или новее):

   ```
   python3 --version
   ```

## Шаг 3. Скачать код проекта на сервер

1. Скачайте (клонируйте) проект в папку `/opt/content-bot` — это
   стандартное место для сервисов, которыми управляет systemd, и путь,
   который уже прописан в `deploy/systemd/content-bot.service`:

   ```
   git clone <ссылка-на-репозиторий> /opt/content-bot
   cd /opt/content-bot
   ```

   Если репозиторий приватный и `git clone` спросит логин/пароль или
   токен — используйте те же данные, что и при работе с репозиторием на
   своём компьютере.

2. Создайте виртуальное окружение внутри папки проекта:

   ```
   python3 -m venv .venv
   ```

3. Установите зависимости бота внутрь этого окружения:

   ```
   .venv/bin/pip install -r requirements.txt
   ```

   Команда должна завершиться без строк `ERROR`.

## Шаг 4. Создать файл `.env` прямо на сервере

**Важно:** файл `.env` содержит настоящий секретный токен бота и ключ
ИИ-сервиса. Его нельзя коммитить в Git (он и так в `.gitignore`, см.
`.env.example`) и **нельзя пересылать через открытые каналы** (обычные
чаты, почту, публичные заметки) — только вводить напрямую на сервере или
передавать через защищённый канал (например, менеджер паролей).

1. Находясь в `/opt/content-bot` на сервере, создайте файл `.env` из
   примера:

   ```
   cp .env.example .env
   ```

2. Откройте файл в текстовом редакторе `nano` (он уже есть в Ubuntu):

   ```
   nano .env
   ```

3. Заполните значения так же, как описано в `README.md` (шаг 5), но здесь
   — уже **настоящими** боевыми данными:

   ```
   BOT_TOKEN=реальный_токен_от_BotFather
   AI_PROXY_API_KEY=реальный_ключ_ии_сервиса
   OWNER_CHAT_ID=ваш_числовой_chat_id
   ```

   Остальные переменные (`DAILY_LIMIT`, `MONTHLY_LIMIT`, `DB_PATH`,
   `LOG_LEVEL` и другие) можно оставить со значениями по умолчанию, как в
   `.env.example` — они уже разумные для старта.

   **`OWNER_CHAT_ID` должен быть реальным Telegram chat id владельца** —
   именно на него бот будет присылать уведомления о падении процесса (см.
   Шаг 6 ниже). Узнать свой chat id можно, написав боту `@userinfobot`.

4. Сохраните файл в `nano`: `Ctrl+O`, затем `Enter`, затем выйдите
   `Ctrl+X`.

5. Проверьте права доступа к файлу, чтобы его не мог прочитать никто, кроме
   владельца сервера:

   ```
   chmod 600 .env
   ```

## Шаг 5. Настроить systemd, чтобы бот работал постоянно

Файл `deploy/systemd/content-bot.service` уже готов в репозитории и
рассчитан именно на путь `/opt/content-bot` — если вы следовали шагам выше
дословно, ничего в нём менять не нужно.

1. Скопируйте файл сервиса в системную папку systemd:

   ```
   cp /opt/content-bot/deploy/systemd/content-bot.service /etc/systemd/system/content-bot.service
   ```

2. Сообщите systemd, что появился новый сервис:

   ```
   systemctl daemon-reload
   ```

3. Включите автозапуск бота при перезагрузке сервера и запустите его прямо
   сейчас — одной командой:

   ```
   systemctl enable --now content-bot
   ```

4. Проверьте, что бот запустился и работает:

   ```
   systemctl status content-bot
   ```

   Вы должны увидеть зелёную строку `active (running)`. Если там написано
   `failed` — смотрите логи следующей командой.

5. Посмотреть свежие логи бота (полезно при любой проблеме) — команда
   показывает новые строки логов в реальном времени, остановить просмотр
   можно `Ctrl+C`:

   ```
   journalctl -u content-bot -f
   ```

6. Проверьте бота вживую: откройте Telegram, найдите своего бота и
   отправьте `/start` — должен прийти приветственный текст. Это значит,
   что бот действительно работает на сервере.

## Шаг 6. Проверить автоматический перезапуск при падении

Это подтверждает требование из `Plan.md`: если процесс бота падает,
сервер должен поднять его заново сам, без участия человека.

1. Принудительно убейте процесс бота (имитация неожиданного падения):

   ```
   systemctl kill content-bot
   ```

   Эта команда отправляет процессу сигнал остановки, как будто бот
   аварийно завершился.

2. Подождите 5–10 секунд (в `content-bot.service` настроен перезапуск
   через 5 секунд после падения), затем проверьте статус:

   ```
   systemctl status content-bot
   ```

   Если в выводе снова `active (running)` с недавним временем запуска —
   автоматический перезапуск работает как задумано. Если сервис остался в
   состоянии `failed` или `inactive` — проверьте логи (`journalctl -u
   content-bot -f`) и убедитесь, что `.env` заполнен без ошибок (частая
   причина: опечатка в `BOT_TOKEN`, из-за которой бот падает при каждом
   старте и systemd в итоге сдаётся после 5 попыток за 5 минут — см.
   `StartLimitBurst` в файле сервиса).

3. Дополнительно можно убедиться, что бот и правда отвечает в Telegram
   после перезапуска — отправьте `/start` ещё раз.

## Шаг 7. Проверить, что уведомление владельцу о падении реально доходит

`systemctl kill` (шаг 6 выше) присылает боту сигнал остановки — это
проверяет, что systemd перезапускает процесс, но не обязательно проходит
через тот же код, что ловит непредвиденную ошибку внутри бота
(`bot/main.py`, `notify_owner`). Чтобы убедиться, что уведомление реально
доходит владельцу в личку, нужно один раз спровоцировать настоящее
необработанное исключение внутри работающего бота.

Проще всего это сделать вместе с разработчиком: временно добавить в код
заведомую ошибку (например, `raise RuntimeError("тест уведомления")` в
начале `run()` в `bot/main.py`), задеплоить эту версию, дождаться падения
и уведомления, затем убрать тестовую ошибку и задеплоить снова. Не
усложняйте эту проверку — она нужна один раз, чтобы убедиться, что канал
уведомлений в принципе работает, а не как регулярная процедура.

Что должно произойти: в Telegram-чате с ботом (в личке владельца, chat id
которого указан в `OWNER_CHAT_ID`) должно появиться сообщение вида:

> Бот остановился из-за непредвиденной ошибки и не отвечает. Сервер должен
> перезапустить его автоматически — если бот не восстановится сам в
> течение нескольких минут, обратитесь к разработчику. Подробности в
> логах на сервере.

Если сообщение не пришло — самая частая причина: `OWNER_CHAT_ID` в `.env`
указан неверно (не тот числовой ID) или владелец никогда не писал боту
`/start` (Telegram не позволяет боту первым написать пользователю, пока
пользователь сам не начал с ним диалог).

## Шаг 8. Финальная проверка: секретов нет в репозитории

Выполните на сервере в папке `/opt/content-bot` (или на своём компьютере
в клонированном репозитории) одну команду:

```
git log -p -- .env
```

Если в ответ ничего не выведено (пустой вывод) — значит, файл `.env`
никогда не попадал в Git-историю, всё в порядке. Если команда что-то
показала — значит, секреты когда-то были закоммичены, и об этом нужно
немедленно сообщить разработчику (потребуется как минимум сменить токен
бота и ключ ИИ-сервиса).

Дополнительно можно проверить командой `git status`, что `.env`
показывается как игнорируемый (не отслеживаемый) файл, а не как
"untracked" файл, который вот-вот можно случайно закоммитить.

## Итог: чек-лист готовности

- [ ] Сервер на Timeweb Cloud создан, оплата в рублях подтверждена
      владельцем.
- [ ] Бот отвечает на `/start` в Telegram при обращении с реального
      аккаунта.
- [ ] `systemctl kill content-bot` → бот поднимается сам в течение
      нескольких секунд (Шаг 6).
- [ ] Уведомление о падении реально дошло в личку владельцу хотя бы один
      раз (Шаг 7).
- [ ] `git log -p -- .env` пустой — секретов в репозитории нет (Шаг 8).
- [ ] Пройден полный смоук-тест из `docs/manual-checklist.md`, раздел
      "Смоук-тест в проде".

Провизия реального сервера, ввод настоящих секретов и сам смоук-тест — это
действия, которые должен выполнить владелец (или технический помощник) на
живом сервере; эта инструкция описывает шаги, но не выполняет их
автоматически.
